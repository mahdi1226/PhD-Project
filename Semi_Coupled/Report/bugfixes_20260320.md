## Fix 1: θ-lagging in magnetic assembler (paper fidelity)

**File:** `core/phase_field.cc`, line 752

**Problem:** The magnetic assembler evaluated χ(θ) at θ^k (just-solved CH value) instead of θ^{k-1} (previous time step). Paper Theorem 4.1 requires all material coefficients at θ^{k-1} for energy stability.

**Change:** `theta_relevant_` → `theta_old_relevant_` in `solve_magnetics()` call to `magnetic_assembler_->assemble()`.

**Impact:** Stability/accuracy, not symmetry. Both sides of domain use the same theta so this doesn't cause left-right asymmetry, but it violates the paper's energy stability condition.

## Fix 2: CH convection — conservative → skew-symmetric form

**File:** `assembly/ch_assembler.cc`, lines 75-76, 85-87, 145-157, 189-194

**Problem:** The Cahn-Hilliard convection used the conservative form `(θU, ∇Λ) = -∇·(θU)·Λ` instead of the paper's skew-symmetric form `B_h(U, θ, Λ) = (U·∇θ, Λ) + ½div(U)(θ, Λ)` (Paper Eq 37, 42a).

The difference is `½div(U)θΛ`. While div(U) is weakly zero (incompressibility enforced weakly), it is NOT pointwise zero with Q1/DG-Q1 velocity-pressure elements. The extra term acts as an artificial mass source/sink at the interface proportional to the local velocity divergence.

**Changes:**
1. Added `update_gradients` to velocity FEValues
2. Added `ux_gradients`, `uy_gradients` storage
3. Computed `div_U = ux_gradients[q][0] + uy_gradients[q][1]`
4. Added `+0.5 * div_U * theta_old_q * Lambda_i * JxW` to RHS

**IBP derivation:**
Paper Eq 42a has `+B_h(U, θ, Λ)` on LHS, which moved to RHS as `-B_h`.
- Skew: `-(U·∇θ, Λ) - ½(divU)(θ, Λ)`
- IBP of first term: `= (θU, ∇Λ) + (θ·divU, Λ) - ½(divU)(θ, Λ) = (θU, ∇Λ) + ½(divU)(θ, Λ)`
- Code had only `(θU, ∇Λ)`, missing the `+½(divU)(θ, Λ)` term.

**MMS impact:** None — MMS velocity is exactly divergence-free, so `½div(U)θ = 0` for MMS. No MMS source update needed.

**Impact:** Affects interface dynamics and wavelength selection. Could contribute to wrong spike count.

## Fix 3: Kelvin force face term — missing plus-side CG test functions

**File:** `assembly/ns_assembler.cc`, lines 753-863 (kernel), 960-976 (AMR case), 1009-1027 (same-level case)

**Problem:** The Kelvin force face integral `-μ₀ ∫_F ([[H]]·{M})(V·n) dS` was only assembled using CG velocity test functions V from the **minus cell**. Plus-cell test functions — which also have support at the face — were completely skipped.

Each interior face is processed once (to avoid double-counting). For DG this is fine because both sides are handled via a combined matrix. But for CG test functions, BOTH cells' test functions have support at the face. The "process once" pattern must include test functions from BOTH sides.

**Why this causes asymmetry:** Which cell is "minus" depends on the cell ID ordering / MPI partition — this is NOT left-right symmetric. So the Kelvin force face contribution was biased toward one side's test functions, breaking the physical left-right symmetry.

**Changes:**
1. Added `ux_fe_face_plus`, `uy_fe_face_plus` (FEFaceValues for plus-side velocity)
2. Added `face_rhs_ux_plus`, `face_rhs_uy_plus`, `ux_dofs_plus`, `uy_dofs_plus`
3. Updated `kelvin_face_kernel` lambda to accept plus-side velocity FEValues
4. Kernel now assembles both minus and plus cell test functions using the SAME face-fixed quantities (normal, [[H]], {M})
5. Both cell contributions are distributed to global RHS
6. Updated both same-level (Case 3) and AMR finer-neighbor (Case 1) call sites

**Impact:** This is the strongest candidate for the asymmetric spike growth. Missing half the Kelvin face integral, with the missing half determined by mesh-dependent cell ordering, directly breaks the simulation's physical symmetry.

## Other changes (non-bug)

### Critical wavelength diagnostic fix
**File:** `core/phase_field.cc`, line 175-182

**Problem:** The diagnostic formula `λ_c = 2π√(σ/(ρg))` used `σ = lambda = 0.05` (the CH parameter), giving λ_c = 0.0077 (~130 spikes). Wrong — the effective surface tension in the phase-field model is `σ_eff = λ/ε = 5.0`.

**Fix:** Changed to `σ_eff = λ/ε`, `ρg_eff = r·g`. Now correctly predicts λ_c = 0.257 (~3.9 spikes), matching the paper's 4 spikes.

### Kelvin force comment fix
**File:** `physics/kelvin_force.h`, line 21

**Problem:** Comment incorrectly said `[[H]]=0` because φ is CG. But H = ∇φ is piecewise (computed element-wise), so `[[∇φ]] ≠ 0` across faces.

**Fix:** Updated comment to `non-zero: H=∇φ is piecewise, [[∇φ]]≠0 across faces`.

### ns_assembler ghosting cleanup
**File:** `assembly/ns_assembler.cc`, line 678-681

**Change:** Cell loop in `assemble_kelvin_force` now uses `phi_ghosted`/`Mx_ghosted`/`My_ghosted` instead of `phi_solution`/`Mx_solution`/`My_solution`. Not a real bug (the inputs were already ghosted vectors), but makes the code consistent — both cell and face loops now use the same locally-named ghosted copies.
